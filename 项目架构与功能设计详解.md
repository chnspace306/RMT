# RMT Analytics — 项目架构与功能设计详解

## 一、项目概述

RMT Analytics 是一个基于随机矩阵理论（Random Matrix Theory, RMT）的高维数据特征值谱分析平台。平台以 FastAPI + Vue 3 全栈架构实现，支持用户上传真实数据集或使用内置示例，通过协方差矩阵的特征值谱分布与理论分布（Marchenko-Pastur 分布、Wigner 半圆律）的对比，自动识别数据中具有统计显著性的"异常信号"，并提供 IPR 局部化分析、RMT 降噪热力图、滚动窗口系统性风险演化监控等高级分析功能。

**技术定位：** 面向金融风控、环境监测、神经科学、生物信息学等领域的高维数据分析工具，为研究者提供一个将随机矩阵理论应用于实际数据探索的可视化平台。

## 三、后端架构详解

### 3.1 模块划分

后端代码组织为三个核心模块，职责单一、低耦合：

**`backend/main.py`（497 行）** — FastAPI 应用主体，包含：
- 应用初始化与 CORS 中间件配置
- `process_dataframe()` 核心分析引擎（约 160 行）
- `run_rolling_analysis_logic()` 滚动窗口分析函数
- `get_datasets_dir()` 数据集目录多路径搜索
- 6 个 API 端点路由

**`backend/schemas.py`（72 行）** — 数据模型层，定义 10 个 Pydantic 模型：
- `WignerRequest` / `WignerResponse` — Wigner 模型请求/响应
- `MPRequest` / `MPResponse` — M-P 模型请求/响应（最复杂，包含 14 个字段）
- `AnalyzeRequest` — AI 分析请求（含特征向量摘要）
- `TheoreticalCurve` — 理论曲线（x/y 坐标数组的通用结构）
- `TopComponent` — 特征向量的单个高权重成分
- `OutlierEigenvector` — 异常特征值及其特征向量完整信息
- `RollingResponse` — 滚动窗口时序数据
- `HeatmapResponse` — 降噪热力图 base64

**`backend/rmt_core.py`（55 行）** — 纯数学函数层，依赖仅 NumPy + SciPy：
- `generate_goe_eigenvalues(n, scale)` — GOE 随机矩阵生成与特征值计算
- `wigner_semicircle_pdf(x, r)` — Wigner 半圆律理论 PDF
- `generate_mp_eigenvalues(n, p, scale)` — 样本协方差矩阵生成与特征值计算
- `mp_pdf(x, q, sigma_sq)` — Marchenko-Pastur 理论 PDF

### 3.2 核心分析引擎设计

`process_dataframe()` 是整个后端最核心的函数，作为统一的矩阵分析入口，被多个端点复用。其处理流水线如下：

```
                     输入: pd.DataFrame
                            │
              ┌─────────────┴─────────────┐
              │  1. 数据清洗               │
              │  - 移除全 NaN 行/列         │
              │  - 计算稀疏度 (sparsity)    │
              └─────────────┬─────────────┘
                            │
              ┌─────────────┴─────────────┐
              │  2. 缺失值填充策略          │
              │  - zero: 填充 0             │
              │  - mean: 列均值填充         │
              │  - drop: 删除含 NaN 行      │
              └─────────────┬─────────────┘
                            │
              ┌─────────────┴─────────────┐
              │  3. 标准化 (可选)           │
              │  z-score: (x-μ)/σ          │
              │  未标准化: 原始协方差        │
              └─────────────┬─────────────┘
                            │
              ┌─────────────┴─────────────┐
              │  4. 协方差矩阵 C = XᵀX/n   │
              │  5. 特征值分解 (eigh)       │
              │     eigenvalues + eigenvectors│
              └─────────────┬─────────────┘
                            │
              ┌─────────────┴─────────────┐
              │  6. MP 理论边界计算         │
              │  λ₊ = σ²(1+√q)²           │
              │  λ₋ = σ²(1-√q)²           │
              └─────────────┬─────────────┘
                            │
              ┌─────────────┴─────────────┐
              │  7. 异常值检测              │
              │  λ > λ₊ → 异常信号          │
              │  提取 Top-5 特征向量成分     │
              └─────────────┬─────────────┘
                            │
              ┌─────────────┴─────────────┐
              │  8. IPR 计算               │
              │  IPRⱼ = Σᵢ (vᵢⱼ)⁴        │
              └─────────────┬─────────────┘
                            │
              ┌─────────────┴─────────────┐
              │  9. RMT 降噪               │
              │  信号子空间重建 → C_clean   │
              │  Matplotlib 并排热力图生成   │
              └─────────────┬─────────────┘
                            │
                     输出: MPResponse
```

**关键设计决策：**
- 使用 `np.linalg.eigh` 而非 `eig`，因为协方差矩阵是对称矩阵，`eigh` 专为 Hermitian 矩阵优化，返回实特征值且计算速度更快
- 标准化时 σ² = 1 是一个重要设计：标准化后的数据协方差矩阵等价于相关系数矩阵，此时 MP 分布的理论方差参数为 1，简化了理论边界的计算
- 异常值检测最多保留 10 个（`if rank >= 10: break`），覆盖主要的信号成分，避免界面信息过载

### 4.2 状态管理与数据流

前端未使用外部状态管理库（如 Pinia 虽已安装但未实际使用），而是采用 Vue 3 组合式 API 的 `ref` 进行组件内状态管理，通过 props/emits 实现父子组件通信。这种设计在单页面应用且组件层级不深的场景下简洁高效。

核心状态变量（30+）：

- **模型与参数**：`currentModel` (MP/Wigner)、`q`、`sigmaSq`、`bins`、`fillStrategy`、`useStandardization`、`normType`
- **分析结果**：`eigenvalues`、`theoreticalCurve`、`lambdaPlus`、`lambdaMinus`、`anomalies`
- **数据集管理**：`currentDataset`、`uploadedDatasets[]`、`fileLinesCache[]`
- **视图控制**：`currentView` (spectrum/ipr/heatmap/rolling)、`expandedEigenvector`、`isAiDrawerOpen`
- **AI 配置**：`aiSettings` (baseUrl, apiKey, modelName)

**数据流向：**

```
用户操作 → App.vue (状态更新) → API 调用 (rmt.ts)
    → 后端处理 → JSON/SSE 响应
    → 更新 eigenvalues / theoreticalCurve 等 ref
    → RmtChart.vue watch → ECharts 重新渲染
    → 异常值 emit → anomalies ref 更新 → 面板渲染
```

**图表 ↔ 面板双向联动机制：**
- 图表 → 面板：`RmtChart` 监听 `getZr().on('click')`，用户单击异常值区域时计算最近异常值索引并 `emit('outlier-click', idx)`，App.vue 设置 `expandedEigenvector = idx`，面板自动滚动至对应异常值
- 面板 → 图表：用户展开异常信号因子面板时，App.vue 传递 `highlightAnomalyIndex` prop 给 RmtChart，图表通过 `dispatchAction({type:'highlight'})` 高亮对应柱形

### 4.3 RmtChart 自适应分箱算法

`createHistogram()` 实现了一个智能分箱算法，核心特点：

1. **动态 bins 范围裁剪**：当数据中存在极端异常值（如 λ > 2×λ₊）时，将 bins 范围裁剪到 λ₊ × 1.2，避免极端值压缩主体分布的可视化效果
2. **异常值尾端池化**：超过 maxIdx 的极端值统一归入最后一个 bin，避免丢失数据
3. **频率/密度双模式**：支持概率密度（Density = count / (N × binWidth)）和原始频数两种归一化模式
4. **异常值自动着色**：超过 λ₊（或 Wigner 下低于 λ₋）的 bin 自动使用橙色→红色渐变，主体使用蓝色渐变

### 4.4 四视图模式设计

**视图一：Spectrum（特征值谱）**
- 默认视图，显示经验特征值直方图与理论分布曲线叠加图
- 红色虚线标记理论边界 λ₊（和 Wigner 的 λ₋）
- 橙色/红色柱形标记异常特征值
- 支持图表导出为 PNG

**视图二：IPR（逆参与率局部化分析）**
- 以散点图展示每个特征值（x 轴）对应的 IPR 值（y 轴）
- 计算公式：IPRⱼ = Σᵢ (vᵢⱼ)⁴，其中 vᵢⱼ 是第 j 个特征向量的第 i 个分量
- 物理含义：IPR 越大表示该模式越"局域化"（集中在少数变量上），IPR ≈ 1/N 表示完全"延展"（均匀分布）

**视图三：Heatmap（RMT 降噪相关性矩阵）**
- 并排展示原始相关性矩阵与 RMT 降噪后的相关性矩阵
- 降噪方法：将 λ ≤ λ₊ 的特征值替换为 bulk 均值，仅保留信号子空间重建协方差矩阵
- 使用 Matplotlib 后端渲染为 PNG base64，前端直接显示

**视图四：Rolling（系统性风险演化）**
- 滑动窗口计算每一时刻的最大特征值 λ₁，绘制时间序列折线图
- 红色渐变面积图强调风险区域
- 内置 ECharts DataZoom 缩放组件，支持时间轴拖动
- 应用场景：金融市场系统性风险监测、环境监测突变检测

### 4.5 领域语义解释系统

平台内置了四个领域的语义解释器，通过 `selectedDomain` 选择器切换，为异常特征值提供领域特定的含义：

- **通用模型**：统计主成分，解释偏离主体分布的统计意义
- **金融市场**：第一异常特征值代表"市场模式"（Market Mode）或系统性风险；次级异常值代表行业板块效应
- **环境监测**：异常值对应突发区域性极端事件（沙尘暴、工业泄漏等）
- **神经科学**：对应大脑网络的同步激活区域或主导神经元活动模式

## 五、RMT 核心数学模型

### 5.1 Wigner 半圆律 (Wigner Semicircle Law)

**适用对象**：高斯正交系（GOE）随机矩阵

**生成算法**：
```
A ~ N(0, σ²) (n×n 独立正态随机矩阵)
H = (A + Aᵀ) / √2  (对称化，保持非对角线方差为 σ²)
特征值 = eigh(H)
```

**理论 PDF**：
```
ρ(λ) = (2 / πR²) × √(R² - λ²),  |λ| ≤ R
其中 R = 2σ√n
```

**物理含义**：描述一大类随机对称矩阵的特征值分布，是不依赖于矩阵具体形式的"普适性"规律。

### 5.2 Marchenko-Pastur 分布 (M-P Law)

**适用对象**：样本协方差矩阵（非方阵 → 方阵映射）

**生成算法**：
```
X ~ N(0, σ²) (n×p 独立正态随机矩阵)
C = (1/n) × XᵀX  (p×p 样本协方差矩阵)
特征值 = eigh(C)
```

**理论 PDF**（q = p/n < 1 时）：
```
λ₊ = σ²(1 + √q)²   (理论上界)
λ₋ = σ²(1 - √q)²   (理论下界)

ρ(λ) = √[(λ₊ - λ)(λ - λ₋)] / (2πqσ²λ),  λ ∈ [λ₋, λ₊]
```

**关键性质**：特征值谱集中在 [λ₋, λ₊] 区间内。任何超过 λ₊ 的特征值都极不可能是纯随机波动造成的，因此被识别为"信号"。

### 5.3 信号检测原理（核心创新点）

RMT 方法的核心优势在于提供了一个**非参数化的信号检测阈值**：

传统的 PCA 方法需要人为设定保留多少个主成分，而 RMT 方法通过理论分布提供了一个统计学意义上的"自动阈值" λ₊。任何大于 λ₊ 的特征值都代表了数据中真实存在的协方差结构，而非随机噪声。

这一特性使 RMT 特别适合：
- 金融协方差矩阵的降噪（去除"噪声子空间"）
- 高维生物数据的信号提取
- 复杂系统的结构性变异检测

### 5.4 IPR 局部化度量

**公式**：IPRⱼ = Σᵢ₌₁ᴾ (vᵢⱼ)⁴

- IPR ∈ [1/p, 1]：越接近 1/p 表示模式越"延展"（delocalized），越接近 1 表示越"局域化"（localized）
- 在物理系统中，局域化通常指示某种"相变"或"有序结构"的出现
- 在金融中，高 IPR 的特征向量往往对应集中在少数资产上的特定风险因子

### 5.5 RMT 降噪算法

```
1. 计算协方差矩阵 C 的全部特征值和特征向量
2. 识别信号子空间：{i | λᵢ > λ₊}
3. 计算噪声子空间均值：λ̄_noise = mean({λⱼ | λⱼ ≤ λ₊})
4. 重建降噪协方差矩阵：
   C_clean = Σ_{i∈signal} λᵢ·vᵢvᵢᵀ + Σ_{j∈noise} λ̄_noise·vⱼvⱼᵀ
5. 将 C_clean 转换为相关系数矩阵（对角线归一化）
```

降噪后的矩阵消除了随机噪声的影响，更准确地反映变量间的真实相关结构。

### 5.6 滚动窗口 λ₁ 演化

**目的**：检测高维系统协方差结构的时序变化

**算法**：
```
for t in [0, n-window_size] step step_size:
    window = data[t : t+window_size]
    C_t = cov(window)
    λ₁(t) = max(eigenvalues(C_t))
```

- λ₁ 的突然上升通常指示系统性风险的增加（在金融中）
- λ₁ 的缓慢漂移可以反映系统结构的渐变过程
- 对比理论边界 λ₊ 可以判断当前系统是否处于"异常状态"

## 七、关键设计决策总结

### 7.1 为什么前后端分离？

- 后端专注数值计算（NumPy/SciPy），Python 生态无可替代
- 前端专注交互可视化（ECharts），JavaScript 生态最优
- RESTful API 使前后端可独立部署、独立扩展

### 7.2 为什么用 eigh 而非 eig？

协方差矩阵是对称矩阵（Hermitian），`np.linalg.eigh` 专为此类矩阵优化，保证返回实特征值且计算复杂度更低。

### 7.3 为什么 Matplotlib 用 Agg 后端？

`matplotlib.use('Agg')` 必须在任何其他 matplotlib 导入之前调用。Agg 是非交互式后端，不需要 GUI 环境，适用于 Docker 容器和无头服务器。此配置是云部署的关键前提。

### 7.4 为什么特征值排序从大到小？

`np.linalg.eigh` 返回从小到大排序。`process_dataframe` 中通过 `np.argsort(eigenvalues)[::-1]` 逆序排列，使得 λ₁ 为最大特征值，这在物理学和金融学中是标准约定。

### 7.5 为什么数据集路径需要多路径探测？

应用需要在三种环境下运行：本地开发（`python backend/main.py`）、Docker 容器（`/app/`）、Zeabur 云端（挂载路径不确定）。多路径探测使代码无需修改即可适配所有环境。

### 7.6 前端 API 基础路径的三态设计

```typescript
const API_BASE_URL = (import.meta.env.MODE === 'production')
  ? ''                                    // 生产：相对路径，由 Nginx 代理
  : (import.meta.env.VITE_API_BASE_URL    // 开发：优先使用环境变量
      ?? 'http://127.0.0.1:8000');       // 兜底：本地默认地址
```

这一设计使同一份前端代码无需任何修改即可在三种环境下正确工作：本地开发（直连 8000）、任意自定义后端（环境变量）、生产部署（Nginx 代理）。
